# NIDS (Network Intrusion Detection System) - Technical Architecture

This documentation outlines the core architecture of our NIDS project. The system is divided into four distinct **Subsystems**. This structure makes it easy to assign different modules to different team members, while showing the doctor exactly how the pieces fit together.

---

## 1. The Dashboard & API Subsystem
**Ideal for:** A team member focusing on Web Development, APIs, and User Authentication.

### `app.py`
**Purpose:** This is the heart of the web dashboard and the central traffic controller.
**Key Responsibilities:**
- **Flask Web Server:** Serves the HTML pages and provides all the REST API endpoints (like `/api/ban_ip`, `/api/update_config`).
- **Real-Time WebSockets:** Uses `flask_socketio` to instantly push live network alerts to the dashboard without requiring the user to refresh the page.
- **Authentication:** Handles the secure login system. It tracks failed password attempts and automatically issues "Dashboard Bans" to prevent brute-force attacks.
- **Background Threading:** It launches the packet sniffer in the background, ensuring the web interface remains fast and responsive while heavy network scanning happens in parallel.

### `config.json`
**Purpose:** The central configuration file for the web server.
**Key Responsibilities:**
- Stores the hashed version (SHA-256) of the administrator's password, ensuring plain-text passwords are never saved on disk.

## 2. The Intrusion Detection Subsystem
**Ideal for:** A team member focusing on Cybersecurity, Networking, and Traffic Analysis.

### `packet_sniffer.py`
**Purpose:** The "eyes" of the system that watch network traffic in real time.
**Key Responsibilities:**
- **Raw Packet Capture:** Uses the `scapy` library to hook into the computer's network interface and capture packets as they arrive.
- **Data Parsing:** Strips down complex raw packets into readable data formats (Source IP, Destination IP, Protocol, Payload Size).
- **Delegation:** Instantly forwards the parsed packet data to the Rule Engine for threat analysis.

### `rule_engine.py` & `rules_config.json`
**Purpose:** The "brain" that detects malicious activity.
**Key Responsibilities:**
- **Port Scan Detection:** Detects when an attacker (a single IP) tries to probe multiple different ports rapidly.
- **DDoS / Flooding Detection:** Flags high volumes of packets originating from a single source in a short timeframe.
- **Payload Anomalies:** Identifies suspicious packets that exceed normal size thresholds.
- **Dynamic Configuration:** Reads from `rules_config.json`, allowing the admin to tweak sensitivity (e.g., changing the trigger threshold from 50 packets to 100 packets) directly from the dashboard.

## 3. The Cryptography & Storage Subsystem
**Ideal for:** A team member focusing on Data Security, Cryptography, and File Management.

### `encryption.py`
**Purpose:** Ensures that all stored logs and ban lists are completely tamper-proof.
**Key Responsibilities:**
- **AES-256 Encryption:** Uses military-grade AES-256 encryption (in CBC mode) to scramble the data.
- **HMAC Integrity:** Uses HMAC-SHA256 signatures. If an attacker tries to open a log file and alter even a single character, the HMAC verification will fail and the system will reject the tampered file.
- **Secure Key Derivation:** Uses PBKDF2 with random salts to safely generate strong encryption keys from the admin's password.

### `logger.py` & `decrypt_exported_logs.py`
**Purpose:** Handles the safe storage and retrieval of network alerts.
**Key Responsibilities:**
- **`logger.py`:** Takes the alerts generated by the Rule Engine, encrypts them via `encryption.py`, and safely writes them to `.bin` files inside the `logs/` directory.
- **`decrypt_exported_logs.py`:** A portable recovery tool. If the admin exports a ZIP file of the logs, they can use this standalone script on a completely different computer to securely unlock and read the logs using the master password.

---

## 4. The Threat Mitigation Subsystem
**Ideal for:** A team member focusing on Operating Systems, Scripting, and Incident Response.

### `firewall_manager.py`
**Purpose:** The ultimate defense mechanism that interacts directly with the Windows Operating System.
**Key Responsibilities:**
- **OS-Level Blocking:** When the admin clicks "Block Device" in the dashboard, this file executes native Windows `netsh advfirewall` commands.
- **Total Isolation:** It creates Inbound and Outbound firewall rules that completely drop all network packets (including Pings) from the attacker's IP address, severing their connection to the machine entirely.

### `unban.py`
**Purpose:** The emergency "break-glass" command-line recovery tool.
**Key Responsibilities:**
- **Account Recovery:** If the administrator accidentally bans their own IP address or forgets their setup, they are locked out of the web UI. 
- **Terminal Unbanning:** Running `python unban.py <IP>` in the terminal bypasses the website entirely. It manually decrypts the ban files, clears the target IP, resets the failed login strike counter back to zero, and deletes the Windows Firewall rules to seamlessly restore access to the system.

## 5. System Dependencies (`requirements.txt`)
**Ideal for:** The entire team to understand the external libraries that power the core engine.

To keep the project lightweight and secure, we only rely on four essential third-party Python libraries. Each was specifically chosen for its role in the system:

### 1. `scapy` (The Networking Engine)
- **Why we use it:** Scapy is a powerful interactive packet manipulation program. We use it inside `packet_sniffer.py` to hook directly into the computer's Network Interface Card (NIC). It allows us to capture live packets off the wire, dissect their layers (IP, TCP, UDP), and extract the raw data needed to feed our intrusion detection rules.

### 2. `cryptography` (The Security Vault)
- **Why we use it:** This library provides cryptographic recipes and primitives to Python developers. We use it extensively in `encryption.py` to implement **AES-256-CBC encryption** and **HMAC-SHA256 signatures**. This ensures that even if an attacker gains access to the server's hard drive, they cannot read our logs or modify the ban lists without the master password.

### 3. `flask` (The Web Framework)
- **Why we use it:** Flask is a lightweight WSGI web application framework. It serves as the foundation for `app.py`. We use it to render the HTML dashboard, handle user login sessions securely, and create the RESTful API endpoints that the frontend JavaScript communicates with. It was chosen because it is fast, modular, and easy to secure.

### 4. `flask-socketio` (The Real-Time Bridge)
- **Why we use it:** Traditional web servers require the user to refresh the page to see new data. In a security context, waiting for a refresh is too slow. We use Socket.IO to upgrade the Flask server to use **WebSockets**. This creates a persistent, two-way connection between the server and the browser, allowing the `packet_sniffer.py` to push live attack alerts to the user's screen the absolute millisecond they occur.
